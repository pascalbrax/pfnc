# pfnc Roadmap

`pfnc` is a Midnight-Commander-style dual-pane file manager for Linux, specialized for local-vs-remote (SSH/SFTP) file management: ultra-compatible with any remote host, extremely stable, multi-threaded, with directory sync and (eventually) an auto-deployed agent for a faster LAN transfer path.

Work is split into three phases, each additive rather than a rewrite of the last. The seams that make that possible — the `Vfs` trait (`crates/core/src/vfs.rs`) and the `Transport` trait (`crates/core/src/transport.rs`) — were built in Phase 1 and Phase 2 specifically so Phase 3 could slot in without touching existing code paths.

## Phase 1 — Local + SFTP dual-pane manager — ✅ Complete

Core file manager: dual-pane TUI, local filesystem browsing, SSH/SFTP remote browsing, file operations, and archive browsing.

- **Filesystem abstraction**: a `Vfs` trait keyed by a `Location` enum (`Local`, `Remote{profile_id}`, `Archive{base, archive_path}`), with streaming read/write and a `capabilities()` query. Panels only ever hold `(Location, VfsPath)` + cached metadata, never backend-specific types.
- **Local backend** (`crates/vfs-local`): `std::fs`-backed.
- **SFTP backend** (`crates/vfs-sftp`): built on `ssh2` (libssh2 bindings) rather than a pure-Rust SSH stack — libssh2 is far more battle-tested against oddball real-world servers (old ciphers/kex, Dropbear, hardened appliance sshd), which matters for the "ultra compatible" goal. TOFU host-key verification against the real `~/.ssh/known_hosts`, never silent-accept. Auth: ssh-agent → key file → password → keyboard-interactive.
- **Archive backend** (`crates/vfs-archive`): `.tar` / `.tar.zst` / `.zip` browsable exactly like a directory tree, zip-slip path-traversal protection, extraction reuses the generic copy path.
- **File operations**: F5 Copy, F6 Move/Rename, F7 MkDir, F8 Delete — all cancellable background jobs with progress bars, thread-per-job (no async in the core app — see "Concurrency model" below), working across local↔remote.
- **F3 Edit**: launches nano→pico→vi on the cursor file, suspending/resuming the TUI cleanly (remote files are downloaded to a temp file and only uploaded back if changed).
- **F9 Connect**: SSH connect dialog with a saved-profile picker (F1–F9), auto-saves successful connections (host/port/username only — passwords are never persisted).
- **F1 Help**: an "About" box (tool description + the project's GitHub URL) plus, when the active panel is remote, connection diagnostics — protocol (SFTP), the remote host's OS, and (Phase 3) whether the QUIC fast path is enabled for that connection and, if so, its local/remote ports and the listening `pfnc-agent`'s PID. See Phase 3's `Vfs::connection_info`.
- **Config** (`crates/config`): `~/.config/pfnc/config.toml` — session restore of last local panel locations, saved connection profiles, configurable F-key bindings, general settings (`confirm_delete`, `show_hidden`, `sync_delete_extraneous`, `enable_quic_fast_path`).
- **UI polish**: directory listings sort folders-before-files then alphabetically; panel entry lists scroll correctly (with PageUp/PageDown) once a directory has more entries than fit on screen; going up a directory restores the cursor onto the folder just left (Midnight-Commander behavior); each panel's border shows a connection-type glyph before its location label — ⌂ local, ⇄ remote/SFTP, or (once Phase 3's QUIC fast path is actually available for that connection) ⚡. `PanelState::quic_available` is set once, right after a connect/archive-open job completes, from `Vfs::fast_transport()`'s already-cached result (probed during that same background job, never on the UI thread) — a static snapshot of that one connection's capability, not a live "is this specific transfer using QUIC" indicator.

**Current F-key layout**: F1 Help · F2 Menu (unimplemented) · F3 Edit · F4 Sync · F5 Copy · F6 Move · F7 MkDir · F8 Delete · F9 Connect · F10 Quit. ("View" was deliberately not given a slot — sacrificed for Sync.)

## Phase 2 — Directory sync — ✅ Complete

**F4 Sync**: rsync-style whole-file directory synchronization. Scans the two panels' current directories, shows a copy/delete summary for confirmation, then applies only what changed — nothing happens before the user approves the plan.

- **Comparison rule**: same size + same mtime isn't good enough on its own (a same-size content change with a stale mtime would be missed), so sync prefers a content hash over mtime whenever both sides can produce one cheaply:
  1. Sizes differ → copy (no hashing needed).
  2. Sizes match → hash both sides (XXH64 via `xxhash-rust`) and compare directly if both succeed.
  3. Either side can't hash → fall back to mtime as a "quick check" (rsync's own classic heuristic).
- **Hashing never costs more bandwidth than it saves**: `LocalFs` hashes in-process (free, no network). `SftpFs` runs `xxhsum` over an SSH *exec* channel — not the sftp data channel — so only a short hash string crosses the wire, never the file's content. If `xxhsum` isn't on the remote host, this gracefully degrades to the mtime fallback (probed once per connection, cached).
- **Type-mismatch handling**: a file replaced by a directory (or vice versa) is handled as remove-then-recreate, not silently skipped or left in an inconsistent state.
- **`sync_delete_extraneous`** config flag (default `false`) controls whether files that only exist on the destination get removed — off by default, so sync is copy-only unless explicitly opted into mirroring.
- **Transport seam**: a `Transport` trait + `negotiate_transport` function (`crates/core/src/transport.rs`) abstract "how bytes actually move for one file transfer" behind an interface that `copy_job` and sync both already go through. Today there's exactly one implementation (`VfsStreamTransport`, the existing SFTP/local stream copy) — this is the seam Phase 3 plugs into.

## Phase 3 — SSH-deployed agent + QUIC LAN fast-path — 🚧 In progress

**Goal**: auto-deploy a lightweight agent to reachable/LAN hosts and negotiate a faster UDP (QUIC-based) transfer path, with automatic SFTP fallback when that isn't possible (host unreachable over UDP, agent deployment fails, older/locked-down host, etc.).

**Why async lives here and nowhere else**: the rest of the app is deliberately thread-per-job with no async runtime (see `crates/core/src/job.rs`) — a single `crossbeam_channel` unifies input, tick, and job-progress events into one event loop. The agent is its own standalone OS process, running on the *remote* host, never linked into the `pfnc` TUI binary — so it's the one place in the whole workspace where `tokio` (required by `quinn`) is appropriate, without that rule being compromised anywhere else.

### Done

- **`crates/agent-linux`** (`pfnc-agent-linux`, binary `pfnc-agent`): a real, working standalone QUIC server proving the fast-path *mechanism*:
  - Ephemeral self-signed certificate generated fresh at agent startup (`rcgen`).
  - A `quinn`/`rustls` QUIC listener using the pure-Rust `ring` crypto backend (deliberately not `aws-lc-rs`, which pulls in a `cmake`+C-toolchain build dependency this project doesn't need).
  - `connect`, a client that pins the *exact* expected certificate byte-for-byte via a custom `rustls` `ServerCertVerifier` (delegating real signature verification to the crypto provider, not reimplementing crypto) — this simulates what real SSH-negotiated cert pinning will eventually do.
  - A real request/response protocol (opcode-prefixed frames, `crates/agent-linux/src/protocol.rs`, `PROTOCOL_VERSION = 2`) on a connection that serves many requests over its lifetime, not just one: `hello` (version exchange), `read_file`/`write_file` — both stream content in bounded `CHUNK_SIZE` (256 KiB) pieces via a plain synchronous callback (`FnMut(&[u8]) -> io::Result<()>` / `FnMut(&mut [u8]) -> io::Result<usize>`, matching `Read`/`Write`'s own shapes) rather than buffering a whole file in memory, on both the agent side (`tokio::fs::File`, streamed directly to/from disk) and the client side. Content has no length prefix — each call opens its own fresh stream and relies on `send.finish()` as the real "done" signal, so neither side needs to know the total size upfront.
  - Proven by integration tests: hello happy path, version-mismatch (connection still succeeds, compatibility flag correctly `false`), wrong-pinned-cert-is-rejected, small and multi-megabyte (spanning many QUIC packets *and* many internal chunks) file round trips, reading a never-written path errors cleanly, and overwriting a path replaces rather than appends.
- **`crates/agent-macos`**, **`crates/agent-bsd`**: placeholder crates (stub `main.rs`, no dependencies) matching the same shape, excluded from the workspace's default build (`[workspace] exclude` in the root `Cargo.toml`) so they don't need to compile on this dev machine — each still builds standalone.
- **SSH-based agent deployment** (`pfnc-vfs-sftp`'s `deploy` module): uploads the already-built `pfnc-agent` binary over the existing SFTP subsystem (mode `0o755`) and execs it over a fresh SSH *exec* channel — the same mechanism used for the `xxhsum` hash check. The agent reports its PID, bound port, and certificate (as hex) on three prefixed startup lines, parsed by `deploy_and_start`; `kill_remote_process` tears it down afterward via a second exec channel. The cert really does travel over the authenticated SSH session here (not a test-only shortcut — that shortcut only ever existed in `agent-linux`'s own in-process handshake tests, which don't deploy anything). Proven end-to-end by an integration test that deploys a real agent onto a real local sshd and performs a real QUIC handshake against it.
- **`QuicTransport` wired into `negotiate_transport`** (`crates/core/src/transport.rs`, `crates/vfs-sftp/src/quic_agent.rs`): a real Copy/Sync between a Local panel and a Remote(SFTP) panel can now actually move bytes over the deployed QUIC agent instead of the SFTP data channel. `pfnc-core` only grew an interface (`Vfs::fast_transport() -> Option<Arc<dyn RemoteFileAgent>>`, `QuicSideTransport`) — it still has no dependency on `quinn`/`tokio`; `SftpFs` is the one real implementation, lazily deploying+connecting on first use and caching the result (success or failure) for the connection's lifetime via a `OnceLock`, same pattern as `quick_hash`'s `xxhsum` probe. Scoped deliberately narrow:
  - Default-on (`enable_quic_fast_path` config flag, default `true`) with a full opt-out; any failure at any stage (OS probe, missing local binary, deploy, handshake) falls back silently to `VfsStreamTransport`, cached so a doomed connection isn't retried per file.
  - Gated by remote OS: a `uname -s` probe over an exec channel must report Linux before any deployment is attempted, since `pfnc-agent-macos`/`pfnc-agent-bsd` are still placeholders.
  - Only the Local↔Remote(SFTP) pair gets a fast path; remote↔remote and archive combinations always use `VfsStreamTransport` — there's no agent-to-agent protocol.
  - `SftpFs::drop` kills the deployed remote agent process so it isn't left running after the connection closes.
  - Proven end-to-end by an integration test driving a real `copy_job` Local→Remote against a real local sshd, asserting the fast path actually connected (not just that it theoretically could) and that the transferred bytes are intact.
- **True streaming for `read_file`/`write_file`**: neither the wire protocol nor `QuicTransport` buffers a whole file in memory anymore (see the `agent-linux` bullet above). `RemoteFileAgent`'s trait methods take a plain `&mut dyn Write`/`&mut dyn Read` instead of a `Vec<u8>`/`&[u8]` blob; `QuicSideTransport` wraps whichever side is local in a small `ProgressIo` adapter that checks cancellation and reports progress on every chunk, so a QUIC-fast-path transfer now reports progress incrementally and honors mid-transfer cancellation, matching `VfsStreamTransport`'s existing chunked behavior instead of only checking/reporting once per whole file.
- **Connection diagnostics in the F1 Help box**: `Vfs::connection_info() -> Option<ConnectionInfo>` (`crates/core/src/vfs.rs`) is a new, purely-diagnostic trait method — plain `String`/`u16`/`u32` fields only, so `pfnc-core` still never depends on `quinn`/`ssh2`. `SftpFs` reports the protocol (`"SFTP"`), the remote OS (from a `uname -s` probe, warmed unconditionally at connect time regardless of `enable_quic_fast_path`), and — when a QUIC agent is actually connected — its local/remote UDP ports and the listening `pfnc-agent`'s PID ("who is listening"). Like `fast_transport`, `connection_info` only ever peeks already-warmed `OnceLock` caches, so opening the F1 dialog never blocks on network I/O. `crates/app/src/app.rs`'s `connection_info_for` resolves this for whichever `Location` the active panel points at, recursing through an archive to its base connection the same way `quic_available_for` already does for the panel glyph.
- **QUIC warm-up runs on its own background job, decoupled from Connect**: `VfsRegistry::connect_and_cache` (`crates/vfs-sftp`) went back to being just SSH connect + SFTP init + the cheap remote-OS probe — it no longer touches `Vfs::fast_transport()` at all, so the Connect dialog's modal Progress job is fast again regardless of whether the fast path ends up usable (deploying and QUIC-handshaking with a real agent takes on the order of a second when it works, and considerably longer when it's failing and hitting timeouts — blocking Connect on that was a real, reported usability problem). Instead, once a Connect job completes and the panel is already browsable, `actions::submit_connect` spawns a second, silent job (tracked in the new `App::background_jobs: HashMap<JobId, BackgroundJob>`, alongside a `BackgroundJob::WarmQuicFastPath` variant) that calls `fast_transport()` on the same cached `SftpFs`. `handle_job_event`'s `JobEvent::Finished` handling checks `background_jobs` before its existing `Mode::Progress`-based dispatch, so this job's completion never touches `app.mode`/`app.status` — it just flips the relevant panel(s)' `quic_available` (via `update_quic_available_for_profile`, recursing through archives the same way) once the probe resolves, whichever way that goes.
- **Real-world deployment/connect bugs found via the new diagnostic logging and fixed**:
  - `deploy_and_start`'s startup-line parsing (`crates/vfs-sftp/src/deploy.rs`) assumed the *first three lines* read from the exec channel were exactly the agent's `PFNC-AGENT-*` lines — some real `sshd`/shell configs write something of their own first (a MOTD, a banner) even for a plain exec channel, which made deployment fail outright. `parse_startup_lines` (now a standalone, unit-tested function) identifies each field by its prefix instead, tolerating and logging (not failing on) anything else, bounded so a remote that never actually produces the expected output still fails promptly rather than hanging.
  - `connect_quic_agent` (`crates/vfs-sftp/src/quic_agent.rs`) built the QUIC socket address via `SocketAddr::parse`, which only ever accepts a literal numeric IP — deterministically failing for any host reachable only by name (DNS, mDNS, `/etc/hosts`), even though the SFTP connection to that same host obviously already worked (`TcpStream::connect` resolves names). Switched to `ToSocketAddrs`, the same resolution mechanism.
  - IPv4/IPv6 address-family mismatch: `pfnc_agent_linux::connect`'s client endpoint hardcoded an IPv4 (`0.0.0.0:0`) bind, which `quinn` correctly rejects connecting from when the resolved remote address is IPv6 (increasingly common on real LANs via SLAAC/ULA) — fixed by picking the bind family to match the target address. The agent's own listener (`crates/agent-linux/src/main.rs`) had the same problem from the other side (bound `0.0.0.0`, unreachable to an IPv6-resolving client) — switched to `[::]`, which is dual-stack by default on Linux. **Confirmed fixed against a real IPv6-only-reachable LAN host** (F1 now shows "QUIC fast path: enabled" there).
  - That `[::]` bind then broke a *different* real host outright (`Error: Address family not supported by protocol (os error 97)`) — some hosts have IPv6 disabled at the kernel level entirely (common on minimal/hardened VPS images), where even attempting an IPv6 bind fails immediately rather than the address just being unreachable. Neither family can be assumed, so `main.rs` now probes with a cheap, throwaway `UdpSocket::bind("[::]:0")` first and only commits to the real (cert-bearing) QUIC bind on whichever family that probe says actually works.
  - `quinn`'s own internal handshake timeout turned out to be tens of seconds — far too long for what's meant to be a quick background probe with a fast SFTP fallback (a real firewalled host was observed stalling ~40s on the handshake, then a further ~15s past that). `connect_quic_agent` now wraps the handshake+hello exchange in an explicit, much shorter `tokio::time::timeout`, and uses `Runtime::shutdown_background()` instead of relying on `Runtime`'s `Drop` (which blocks waiting for any of `quinn`'s own lingering background tasks to wind down) on every failure path. Caught a real bug of its own while fixing this: `tokio::time::timeout(..)` must be constructed *inside* a running runtime (it looks up the reactor immediately, not lazily) — building it as a bare argument to `block_on` from sync code panics with "there is no reactor running." The existing real-agent integration test caught this immediately once run, before it shipped.

### Known limitations

- **The background QUIC warm-up job can make a Linux host feel slower to use right after connecting, even though Connect itself is no longer blocked by it.** `ssh2::Session` (confirmed via the crate's own source/docs) serializes *all* operations from one session through an internal mutex — so while the background job uploads `pfnc-agent` over the same session's SFTP subsystem and execs it, any regular browsing operation on that connection queues up behind it. This is worse in a debug build, where the uploaded binary is tens of MB; a release build (`cargo build --release`) should shrink it dramatically and make the effect much less noticeable. The properly-decoupled fix (a separate SSH session dedicated to the background deploy) is deliberately not done yet — for password auth it would mean holding the password in memory somewhat longer than today, a real security tradeoff to decide deliberately rather than bundle in silently.
- **Deployment fails against a remote host whose glibc is older than the one `pfnc-agent` was built against** — confirmed in the field: `/tmp/pfnc-agent: /lib/x86_64-linux-gnu/libc.so.6: version 'GLIBC_2.34' not found` on a real VPS, because the dev machine building the agent had a newer glibc. This isn't a bug in the deploy logic (which already tolerates/logs the failure and falls back to plain SFTP cleanly) — it's the standard consequence of shipping a dynamically-linked `x86_64-unknown-linux-gnu` binary to an arbitrary remote Linux host. The real fix is building `pfnc-agent` statically against `musl` (`x86_64-unknown-linux-musl`) instead, which has zero runtime dependency on the target's glibc version — deliberately not done yet since it needs a musl Rust target + linker set up and verified (`ring`/`rustls`/`quinn`/`tokio` all need confirming they build cleanly under musl), which no environment used so far in this project has had available.

### Not done yet

- **Picking the right prebuilt binary for the remote host's actual OS/architecture** (and, per the glibc finding above, its libc version) — today's deployment (both the standalone test and the `QuicTransport` wiring) only proves same-machine, same-arch, same-glibc-version Linux deployment. A real multi-arch/OS/libc story needs the macOS/BSD agents built below, a musl static build (see "Known limitations"), plus a way to pick the matching binary for a given remote host.
- **True block-level delta transfer**: real rsync-style byte-diffing within a file only pays off once the agent can compute a content signature *on the remote host* without shipping the file over the wire first — the streaming refinement above makes this the next natural piece to build.
- **macOS/BSD agent implementations**: currently placeholders only: no QUIC code, no OS-specific deployment considerations (code signing/notarization on macOS, `pfnc-agent` binary layout on BSD, etc.) worked out yet.
- **Separate SSH session for the background QUIC deploy**, to eliminate the session-contention limitation above, once the credential-lifetime tradeoff for password auth is deliberately decided.

## Explicitly out of scope for now

- Cross-compilation setup for macOS/BSD builds of the agent.
